# Visual Assets

The repository tracks every visual asset the game may draw so art,
metadata, and runtime usage do not drift apart. Anima Echo is a 2D
top-down pixel prototype, and new static art is **vector-source first**:
SVG is the editable source, PNG is the Godot-compatible alpha export.

The companion inventory is
[`docs/visual_assets/inventory.md`](visual_assets/inventory.md). Update it
whenever an asset is added, removed, re-graded, or wired into runtime.

## Asset Roots

- `assets/mine/`, `assets/town/`, `assets/props/`, `assets/ui/` are
  runtime candidate roots. A scene or script may load these paths only
  after the inventory row and sidecar pass review.
- `assets/development/` is the planned early-development visual library.
  Assets here will be generated, reviewable, and usable as future
  replacement sources, but **current runtime code must not load them**.
  As of this revision, no files exist under `assets/development/` yet;
  all planned assets are listed as `todo` in the inventory.

## Hard Rules

1. Every new static asset has an SVG vector source and PNG alpha export.
2. Every PNG has a sidecar metadata file named `<asset>.png.meta.md`.
3. The sidecar must record `vector-source`, `runtime-export`,
   `intended-use`, `license`, and `status`.
4. `license: TBD` is rejected.
5. A code change that adds a new `load()` or `preload()` must review the
   sidecar before loading the asset.
6. `development` assets are not placeholders and are not implemented
   runtime art. They are front-loaded production art sources for review
   and later integration.

## Naming Convention

Runtime assets keep the existing convention:

```text
assets/<category>/<sub-category>/<name>[_<state>][_variant].png
```

Development assets keep the target runtime name without a placeholder or
candidate suffix:

```text
assets/development/<future-category>/<sub-category>/<name>.svg
assets/development/<future-category>/<sub-category>/<name>.png
assets/development/<future-category>/<sub-category>/<name>.png.meta.md
```

Examples:

- `assets/development/ui/icons/copper_nugget.svg`
- `assets/development/ui/icons/copper_nugget.png`
- `assets/development/ui/icons/copper_nugget.png.meta.md`

## Status Legend

- `implemented` - file exists, is loaded by the game, the visual is final
  enough for the current build, and the sidecar is complete.
- `development` - file exists under `assets/development/`, has SVG/PNG/meta,
  and is ready for art review or future wiring, but is not loaded by
  runtime.
- `placeholder` - file exists and is loaded, but is a temporary stand-in
  that should be replaced.
- `todo` - file does not exist yet. The row is the work order.
- `obsolete` - file is no longer used by the game but is kept for reference.

## Sidecar Metadata

Required fields for `implemented`, `development`, and `placeholder` PNGs:

- `id`
- `category`
- `sub-category`
- `source`
- `vector-source`
- `runtime-export`
- `license`
- `status`
- `width`
- `height`
- `palette`
- `description`
- `intended-use`
- `style-notes`
- `created-by`
- `last-reviewed-by`
- `last-reviewed-on`

`implemented` and `obsolete` sidecars also require `audit-on`.
`placeholder` and `obsolete` sidecars require `replacement` when a known
replacement exists.

## Style Guide

- Target mood: detailed cozy top-down pixel RPG art with crisp silhouettes,
  material highlights, earthy mine colors, warm town surfaces, parchment and
  wood UI, and transparent backgrounds.
- Use committed SVG/vector sources. PNG exports should preserve hard pixel
  edges and alpha transparency.
- Use `TEXTURE_FILTER_NEAREST`; do not bake bilinear filtering into source
  images.
- UI icons should remain readable when downsampled to 32x32 or 48x48.
- Larger development art may use review/source dimensions. Record the exact
  width and height in metadata before any runtime integration.

## Development Generation Workflow

> **The generation script does not exist yet.** The workflow below is the
> target design for when the script and assets are created. All 127
> planned development assets are currently `todo`.

The development library will be generated by
`scripts/tools/generate_development_assets.py` _(not yet created)_.

The script is planned to emit:

- SVG vector sources.
- PNG alpha exports.
- Metadata sidecars.
- `assets/development/manifest.json`.
- Temporary contact sheets under `tmp/development_asset_contact_sheets/`.

When an asset graduates into runtime, copy or export it into the appropriate
runtime root in a separate PR, review the sidecar, update the inventory row
from `development` to `implemented`, and only then add the `load()` or
`preload()` reference.

## Review Checklist Before Runtime Loading

1. Inventory row exists and status matches the target use.
2. SVG, PNG, and metadata sidecar exist.
3. Sidecar has all required fields and `license` is not `TBD`.
4. `status` is not `development` if code is about to load the runtime path.
5. Intended use matches the scene/UI code that will load it.
6. Resolution and import scale match the runtime target.
7. Visual review confirms the asset is not too noisy, too flat, or off-style.
